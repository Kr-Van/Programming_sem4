# Задание по SOLID

## Принцип единственной ответственности (SRP — Single Responsibility Principle)

### Нарушение принципа

Рассмотрим пример из предметной области мобильного приложения для фитнеса и здорового образа жизни.

В системе спроектирован класс `WorkoutTracker`, который берет на себя выполнение слишком большого пула разнородных задач:
- хранит параметры текущей спортивной тренировки (длительность, список упражнений);
- рассчитывает объем сожженных пользователем калорий по математической формуле;
- генерирует визуальный график прогресса для отображения на экране смартфона;
- осуществляет прямую отправку данных для сохранения в удаленную базу данных.

```python
class WorkoutTracker:
    def __init__(self, duration_minutes, exercises):
        self.duration_minutes = duration_minutes
        self.exercises = exercises  # Список упражнений и подходов

    def calculate_calories_burned(self):
        # Базовый расчет: 7 калорий за каждую минуту тренировки
        return self.duration_minutes * 7

    def draw_progress_chart(self):
        calories = self.calculate_calories_burned()
        print(f"Отрисовка красивого графика на экране: сожжено {calories} ккал")

    def sync_with_cloud_db(self):
        print("Подключение к удаленному серверу...")
        print(f"Данные тренировки ({self.duration_minutes} мин) успешно сохранены в облако.")
```

## Почему здесь нарушен SRP

Фундаментальная суть принципа SRP заключается в том, что у любого разрабатываемого класса должна быть всего одна-единственная причина для изменения его исходного кода. Класс должен решать только одну изолированную бизнес-задачу.

В текущем примере класс `WorkoutTracker` нарушает это правило, совмещая в себе четыре совершенно разные зоны ответственности:
1. Хранение бизнес-сущности и структуры тренировки.
2. Алгоритмические вычисления спортивных показателей.
3. Логика интерфейса пользователя (UI) и отрисовка графических элементов.
4. Инфраструктурное сетевое взаимодействие и сохранение данных в облако.

Если в будущем команда разработки решит обновить дизайн мобильного приложения и переделать внешний вид графиков, или если изменится API облачного сервера, программисту придется лезть внутрь класса `WorkoutTracker` и менять его код. Это нарушает стабильность системы, так как случайная опечатка в коде графики может полностью сломать базовые расчеты калорий или логику сохранения тренировок.

Подобное нарушение SRP приводит к следующим негативным последствиям:
- Продукт становится жестким, его тяжело развивать и масштабировать.
- Модульное тестирование (Unit-testing) усложняется, так как класс завязан на внешнюю сеть и графический вывод.
- Повышается риск появления скрытых багов в критически важных вычислениях при изменении второстепенных функций.

# Исправление нарушения

Для исправления архитектурной ошибки выполним декомпозицию. Разделим обязанности исходного монолитного класса между четырьмя специализированными программными компонентами.

```python
class Workout:
    def __init__(self, duration_minutes, exercises):
        self.duration_minutes = duration_minutes
        self.exercises = exercises

    def calculate_calories_burned(self):
        return self.duration_minutes * 7


class WorkoutChartVisualizer:
    def draw_chart(self, workout):
        calories = workout.calculate_calories_burned()
        print(f"Отрисовка красивого графика на экране: сожжено {calories} ккал")


class CloudSyncService:
    def sync_data(self, workout):
        print("Подключение к удаленному серверу...")
        print(f"Данные тренировки ({workout.duration_minutes} мин) успешно сохранены в облако.")
```

# Комментарий к исправлению

После проведения рефакторинга архитектура приложения стала чистой и гибкой благодаря строгому распределению ролей:
- Класс `Workout` превратился в чистую бизнес-модель, которая знает всё о параметрах тренировки и умеет считать калории.
- Класс `WorkoutChartVisualizer` изолировал внутри себя всю работу с пользовательским интерфейсом и экраном.
- Класс `CloudSyncService` стал единственным модулем, отвечающим за работу с сетью, протоколами связи и удаленной базой данных.

Теперь каждый из созданных классов решает ровно одну узкую задачу, что на 100% соответствует требованиям принципа SRP.

Преимущества обновленной архитектуры:
- **Высокая гибкость:** Мы можем полностью переписать сетевой слой (например, сохранять данные локально на телефон, а не в облако), абсолютно не трогая код расчета калорий и отрисовки графиков.
- **Изолированное тестирование:** Логику расчета калорий в классе `Workout` теперь можно протестировать мгновенно, без необходимости имитировать подключение к интернету или отрисовывать окна.
- **Безопасность изменений:** Изменения в одном модуле больше не могут случайно сломать работоспособность другого.
